# Feature Spec: Conventional Commits

**Date:** 2026-03-24
**Branch:** `feat/conventional-commits`
**Status:** Draft

## Summary

Add optional Conventional Commits support to ProGen so that auto-commits
generated by `ProGen.Script.action/3` and `ProGen.Script.command/2` produce
commit messages that conform to the Conventional Commits specification.

## Motivation

Some repositories require commit messages to follow the Conventional Commits
standard (e.g., `feat(scope): description`). Currently, ProGen's auto-commit
feature produces messages like `[ProGen] <description>` with no structured
type or scope. Users working in CC-enforced repos cannot use ProGen's
auto-commit without manually rewriting every commit message afterward. By
making CC support opt-in via a project configuration file and per-action
metadata, ProGen can generate compliant commit messages automatically.

## Requirements

1. **Project configuration file:** A repository can optionally include a
   `.progen.yml` (or `.progen.yaml`) file at the project root. This file is
   parsed at runtime to configure ProGen's behavior.

2. **`use_conventional_commits` option:** The configuration file accepts a
   single boolean option `use_conventional_commits` (default: `false`). When
   `true`, ProGen formats all auto-commit messages using the Conventional
   Commits structure: `<type>(<optional scope>): <description>`.

3. **`@commit_type` module attribute for actions:** Each Action module can
   optionally define a `@commit_type` module attribute containing a string.
   The string specifies the CC type and optional scope using parentheses
   per the CC standard (e.g., `@commit_type "docs"` or
   `@commit_type "feat(web framework)"`).

4. **Default `@commit_type` for actions:** If an Action module does not define
   `@commit_type`, the default value is `"chore(action)"`.

5. **Commit type for commands:** Commits generated by
   `ProGen.Script.command/2` use the commit type `"chore(command)"`.

6. **Commit message formatting:** When `use_conventional_commits` is `true`,
   the commit message is formatted as `<commit_type>: [ProGen] <description>`
   where `<description>` is the first argument passed to `action/3` or
   `command/2`. When `false`, the existing `[ProGen] <description>` format is
   preserved.

7. **Configuration module:** Provide a module (e.g., `ProGen.Config`) that
   reads and parses the `.progen.yml` / `.progen.yaml` file, returning a map
   of configuration values. If the file does not exist, return default values.
   If the file is malformed, return a clear error.

8. **YAML dependency:** Add a YAML parsing library to the project dependencies
   to support reading `.progen.yml` files.

## Acceptance Criteria

- When `.progen.yml` contains `use_conventional_commits: true` and an action
  with `@commit_type "feat"` runs via `action/3`, the commit message is
  `feat: <description>`.
- When `.progen.yml` contains `use_conventional_commits: true` and an action
  with `@commit_type "docs(api)"` runs, the commit message is
  `docs(api): <description>`.
- When `.progen.yml` contains `use_conventional_commits: true` and an action
  with no `@commit_type` defined runs, the commit message is
  `chore(action): <description>`.
- When `.progen.yml` contains `use_conventional_commits: true` and a
  `command/2` call runs, the commit message is
  `chore(command): <description>`.
- When `.progen.yml` is absent or `use_conventional_commits` is `false`, the
  commit message remains `[ProGen] <description>` (existing behavior).
- When `.progen.yml` exists but is malformed, a clear error is returned.
- The `ProGen.Config` module returns default values when no config file is
  found.
- All existing tests continue to pass.

## Out of Scope

- TOML configuration file support (YAML only for now).
- Additional configuration options beyond `use_conventional_commits`.
- Conventional Commits body or footer sections.
- Validation of `@commit_type` values against a fixed list of allowed types.
- Breaking change indicators (the `!` suffix or `BREAKING CHANGE` footer).
- Interactive prompts to select commit type at runtime.
