# Implementation Plan: Conventional Commits

**Spec:** `_spec/features/260324_conventional-commits.md`
**Generated:** 2026-03-24

---

## Goal

Add opt-in Conventional Commits support so that auto-commits generated by
`ProGen.Script.action/3` and `ProGen.Script.command/2` produce messages like
`feat(scope): [ProGen] description` when enabled via a `.progen.yml` project
configuration file.

## Scope

### In scope
- `.progen.yml` / `.progen.yaml` project configuration file (YAML)
- `ProGen.Config` module to read, parse, and cache the configuration
- `use_conventional_commits` boolean option (default `false`)
- `@commit_type` module attribute on Action modules (default `"chore(action)"`)
- `"chore(command)"` commit type for `Script.command/2`
- Commit message formatting: `<commit_type>: [ProGen] <description>` when CC enabled
- Validation of `@commit_type` against an allowed-types list
- YAML parsing dependency
- Tests for all new behavior

### Out of scope
- TOML configuration support
- Additional configuration options beyond `use_conventional_commits`
- CC body/footer sections
- Breaking change indicators (`!` suffix, `BREAKING CHANGE` footer)
- Interactive prompts to select commit type at runtime

## Architecture & Design Decisions

### 1. YAML library choice
Use `yaml_elixir` — the most popular and well-maintained YAML library for Elixir. It provides `YamlElixir.read_from_file/1` which returns `{:ok, map}` or `{:error, reason}`, fitting ProGen's error-handling conventions cleanly.

### 2. `@commit_type` capture via `__before_compile__`
The `ProGen.Action.__before_compile__/1` macro already extracts `@moduledoc`
and injects `name/0` and `description/0`. We extend it to also read
`@commit_type` (via `Module.get_attribute/2`) and inject a `commit_type/0`
function. If the attribute is not set, `commit_type/0` returns the default
`"chore(action)"`. This is consistent with how `description/0` is already
generated.

The `@commit_type` should be validated against CC standards (eg type can be one of [feat, fix, build, chore, ci, docs, style, refactor, perf, revert, test])

### 3. `ProGen.Config` as a standalone module

A new `ProGen.Config` module handles file discovery (`.progen.yml` then
`.progen.yaml`), parsing, and validation. It returns a map with known keys
(initially just `use_conventional_commits`). Unknown keys are ignored.
Malformed YAML returns `{:error, reason}`.

### 4. Commit message formatting in `auto_commit/2`

The private `auto_commit/2` function in `ProGen.Script` is the single integration point. It already constructs the commit message. We modify it to:
1. Read config via `ProGen.Config`
2. If CC is enabled, look up the commit type (passed through from the caller)
3. Format as `"<commit_type>: [ProGen] <desc>"` or fall back to `"[ProGen] <desc>"`

### 5. Passing commit_type from action/3 and command/2

- `action/3` resolves the action module and calls `mod.commit_type()` to get the type
- `command/2` uses the hardcoded string `"chore(command)"`
- Both pass the commit type to `auto_commit/3` (expanded from `/2`)

### 6. Parentheses for scope (CC standard)

The feature spec uses parentheses `feat(scope)` which aligns with the Conventional Commits specification. The earlier design doc used brackets `feat[scope]` — we follow the spec file (parentheses).

## Implementation Steps

1. **Add `yaml_elixir` dependency**
   - Files: `mix.exs`
   - Details: Add `{:yaml_elixir, "~> 2.11"}` to the `deps/0` list. Run `mix deps.get`.

2. **Create `ProGen.Config` module**
   - Files: `lib/pro_gen/config.ex`
   - Details: Implement the following public functions:
     - `read/0` — Searches CWD for `.progen.yml` then `.progen.yaml`. If found, parses with `YamlElixir.read_from_file/1`. Returns `{:ok, map}` with defaults merged, or `{:error, reason}` on malformed YAML. If no file exists, returns `{:ok, %{use_conventional_commits: false}}` (defaults).
     - `use_conventional_commits?/0` — Convenience that calls `read/0` and returns the boolean value. Returns `false` on error or missing file.
   - Key behaviors:
     - The config key in YAML is the string `"use_conventional_commits"`. Convert to atom key in the returned map.
     - Non-boolean values for `use_conventional_commits` are treated as an error.

3. **Extend `ProGen.Action.__before_compile__` to capture and validate `@commit_type`**
   - Files: `lib/pro_gen/action.ex`
   - Details: In the `__before_compile__/1` macro, read `Module.get_attribute(env.module, :commit_type)`. If `nil`, default to `"chore(action)"`. Inject a `commit_type/0` function that returns the value. Add a `@callback commit_type() :: String.t()` to the behaviour and include a default implementation in `__using__/1` (returning `"chore(action)"`) so it's overridable. The `__before_compile__` injection overwrites with the actual attribute value if present.

4. **Expand `auto_commit` to accept and use commit type**
   - Files: `lib/pro_gen/script.ex`
   - Details:
     - Change `auto_commit/2` to `auto_commit/3` with signature `auto_commit(desc, commit_type, opts)`.
     - Inside, call `ProGen.Config.use_conventional_commits?/0`.
     - If `true`, format message as `"#{commit_type}: [ProGen] #{desc}"`.
     - If `false`, keep existing `"[ProGen] #{desc}"`.
     - Update `action/3` callers (both atom and string variants) to resolve the action module and call `mod.commit_type()`, passing result to `auto_commit/3`.
     - Update `command/2` to pass `"chore(command)"` as the commit type.

5. **Add tests for `ProGen.Config`**

     - `.progen.yaml` alternate extension works
     - `.progen.yml` takes precedence over `.progen.yaml` when both exist
     - Malformed YAML → `{:error, _}`
     - Non-boolean value for `use_conventional_commits` → `{:error, _}`

## Edge Cases & Risks

- **CWD changes**: `ProGen.Config.read/0` reads from the current working directory. If `Script.cd/1` has been called, the config file lookup follows the new CWD. This is correct behavior — the config should come from the project being generated, not ProGen's own directory. Document this.
- **Config caching**: The config file should be read fresh each time (not cached) since scripts may `cd` between projects. Keep `read/0` simple and uncached for now.
- **Module attribute timing**: `@commit_type` is a compile-time attribute. It must be set *before* `use ProGen.Action` or at module level after `use` but before compilation completes. The `__before_compile__` hook reads it at the right time (after all module-level code has been evaluated). This matches how `@moduledoc` is already handled.
- **Atom key vs string key in YAML**: `YamlElixir` returns string keys by default. The config module must convert `"use_conventional_commits"` → `:use_conventional_commits`.
- **Action module resolution in Script.action/3 (atom variant)**: When called with a module atom, the module is already known. When called with a string name, we need to resolve the module first via `ProGen.Actions.action_module/1`. Both paths must extract `commit_type/0`.
- **Existing tests**: The config file does not exist in any test temp directories, so all existing auto-commit tests will continue to get `false` from `use_conventional_commits?/0`, preserving current behavior.

## Testing Strategy

- **Unit tests** for `ProGen.Config`: file discovery, parsing, defaults, error handling (Step 5)
- **Unit tests** for `commit_type/0`: default value, custom values, scope syntax (Step 6)
- **Integration tests** for end-to-end commit message formatting via `Script.action/3` and `Script.command/2` with CC enabled/disabled (Step 8)
- **Regression**: Run full `mix test` suite to verify no existing tests break (Step 9)
- All tests use isolated temp directories with setup/teardown (existing pattern)
